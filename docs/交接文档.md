## 1. 项目目标与当前边界

这是一个 A 股日线、中低频、paper-only 的量化交易系统原型，目标是从数据同步、回测、paper trading、日报、策略研究，逐步演进到可接入 LLM 的工程化架构。

当前硬边界：
- 不接实盘券商
- 不允许 LLM 进入下单、仓位、风控 veto 控制面
- `RiskAgent` 保持确定性一票否决
- `ExecutionAgent` 只执行已批准订单
- 交易日历必须是真实 A 股交易日历，不用工作日伪造

## 2. 当前已实现的系统结构

当前主链路：

```text
DataAgent -> RegimeAgent -> SignalAgent -> PositionAgent -> RiskAgent -> ExecutionAgent -> MonitorAgent
                         \-> MetaAgent 负责编排和安全降级
```

当前已经完成：
- A 股 Universe、交易日历、数据同步、Parquet 落盘
- 日线事件式回测与 paper trading
- ETF 动量 + 主板突破两个基线策略
- Stage 1-5 多 Agent 演进骨架
- 日报输出：Markdown / JSON / manual orders CSV / data sync report
- Phase 1 LLM：日报只读解释层
- Phase 2 LLM：离线策略研究只读解释层

## 8. 推荐的接手顺序

### 如果接手的是架构线程
建议按下面顺序继续：

1. 先处理 Git 远端同步问题
   - 确认 Ubuntu SSH key 是否已加入 GitHub
   - 把本地 `main` 和 `arch/llm-foundation` 推到远端
   - 之后再恢复“每次改动都 push”的工作流

2. 清理并行线程边界
   - 先确认当前未提交文件属于谁
   - 不要把策略线程改动混入架构线程提交

3. 推进 LLM Phase 3 之前，先补审计与文本缓存契约
   - 文本原文缓存
   - source / timestamp / provider / prompt_hash
   - 输出审计

4. 再考虑把文本特征引入研究层
   - 先进入 report / diagnostics
   - 不要直接喂给 SignalAgent

### 如果接手的是策略线程
建议按下面顺序继续：

1. 只改以下范围：
- `src/quant_system/features/*`
- `src/quant_system/strategies/*`
- `configs/strategy.toml`
- `tests/*strategy*`

2. 优先补齐：
- ETF 策略真实可解释指标
- 主板突破策略成交量/流动性过滤
- `strategy_diagnostics.json` 的打分解释

3. 不要改：
- `RiskAgent` 的 veto 语义
- `ExecutionAgent` 的 approved-only 边界
- `daily_pipeline.py` 的交易控制路径

## 9. 当前最大风险

当前最大风险不是模型，而是工程协作：
- 工作树混有并行线程未提交内容
- 远端 Git 还没形成真实可回滚镜像
- 如果继续在脏工作树上混合提交，后续会很难回滚

因此真正的首要任务是：
1. 把并行线程改动拆清楚
2. 修通 push 或完成远端快照
3. 再继续大规模演进
